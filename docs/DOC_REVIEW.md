The agent layer stays on PydanticAI; the orchestration layer moves.

This keeps the forward-looking scaling story WITHOUT contradicting the project's "no LangChain" stance.

## 🟠 HIGH — should fix before submission

### H1 — "Satisfies 21 CFR Part 11 traceability requirements" is unsupported

Same issue as slide 11 of the PPTX. The claim is strong; the doc provides no supporting detail on WHICH Part 11 subsections are satisfied. A regulatory-literate interviewer will push.

**Fix:** Either soften or specify. Softer version:

> The audit trail design aligns with 21 CFR Part 11 §11.10 principles: secure computer-generated audit trails with time-stamps (§11.10(e)), operational system checks for permitted sequencing via the pipeline FSM (§11.10(f)). Authority checks (§11.10(h)) are a production extension — the prototype's HITL approval is anonymous and would require SSO/LDAP integration for compliance.

This is honest, specific, and pre-empts the follow-up question.

### H2 — HITL Override atomicity is overstated

Current doc:

> on success the DAG node, audit trail, and `FeedbackRow` are all **committed in one transaction**

"In one transaction" implies atomic DB commit. Looking at the Phase 16.2b code path (`OverrideService`, `workflow_hitl.py`), the sequence is:

1. Validate code syntax
2. Execute code on `derived_df`
3. Mutate DAG node
4. Append audit record
5. Commit FeedbackRow to DB

Steps 2-4 touch in-memory state; only step 5 is a DB transaction. If step 3 or 4 raises mid-flight, the in-memory state is in an inconsistent partial-update state. This is NOT a true single transaction — it's a best-effort sequential update with no rollback.

**Fix:** Soften to:

> on success the DAG node, audit trail, and `FeedbackRow` are updated together; on code execution failure the dialog shows the 400 response inline and the original code is preserved (the derived_df mutation is scoped to a try block — failures short-circuit before state mutation).

### H3 — Unverified "3 separate gates in enterprise mode" claim

Trade-offs table says:

> 3 separate gates in enterprise mode for 21 CFR Part 11 --- same engine, different pipeline YAML

This requires `config/pipelines/enterprise.yaml` to actually declare 3 `hitl_gate` steps. I did not verify this during this review (out of scope for the docx read). Before submission, grep the file and confirm:

```bash
grep -c "type: hitl_gate" config/pipelines/enterprise.yaml
```

If it returns 3, the claim holds. If it returns 0 or 1, fix the doc to match reality OR update `enterprise.yaml` to match the doc.

### H4 — Document length probably exceeds HOMEWORK.md §8.3's "2-4 pages"

Extracted markdown is ~500 lines pre-images. In rendered Word format with 4 embedded diagrams + 4 tables, this is probably **5-6 pages**, maybe 7. HOMEWORK.md §8.3 says **"2-4 pages"**.

A reviewer who stops at page 4 misses the Trade-offs and Appendix. Worse: it signals "can't follow instructions".

**Fix options:**

1. **Trim** — move the Data Security section to the body (it's the strongest pharma-specific slide) and move Production Path + Limitations to a separate `PRODUCTION_DESIGN.md` file. HOMEWORK.md §10 calls out production design as optional for Lead candidates, so splitting makes sense.
2. **Compress** — tighten the prose in System Architecture and Memory Design sections. The Agent/module table can lose the "Output Type" column (redundant with the detailed sections below).
3. **Keep as-is + acknowledge** — if you prefer depth over strict length compliance, add a one-line note at the top: "Length: 5 pages including appendix. Appendix sections (Data Security, Production Path, Limitations) are optional reading." This is a calculated risk — some reviewers will respect the thoroughness; some will dock you for not following instructions.

**Recommendation:** Split Production Path + optional Cloud/Deployment sections into a separate file (`docs/PRODUCTION_DESIGN.md`), keep the main doc at 3-4 pages for the core deliverable. This maps cleanly to HOMEWORK.md's two-tier ask (§8.3 for the 2-4 page main + §10 for optional production design).

### M5 — Memory Design section should name the "Bug #5" context

The 3-tool LTM architecture with authority hierarchy is correctly described, but doesn't mention that it was closed post-code-review. Panel interviewers love "I caught a bug in my own design" stories. Add:

> Phase 17.1 (Bug #5 closure): the LTM read loop was initially one-dimensional — only `query_patterns` surfaced to the coder. Post-submission code review revealed that `feedback` and `qc_history` tables were write-only telemetry with no reader. Three separate tools with explicit authority ranking (human > debugger > prior agent) replaced the single-tool approach so the LLM can reason about provenance instead of flat-weighted evidence.

This turns "we have LTM" into "we iterated on LTM after catching our own overclaim" — much stronger.

### M6 — "PydanticAI passed all five orchestration patterns in prototype validation" is unsupported

"Within 0.01s" is specific enough to feel like a real measurement, but there's no citation. If this came from a `prototypes/` test that got deleted, soften to "validated in benchmark tests during the Day 1 framework evaluation". If it came from a real file, cite it.
